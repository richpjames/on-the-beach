# Compile the iOS app (incl. Share Extension) on PRs

**Date:** 2026-07-10
**Status:** implemented

## Problem

The native iOS app is a Capacitor shell whose real payload is a hand-authored
**Share Extension** (`native/ShareExtension/ShareViewController.swift`). That
Swift is the code that actually changes, yet nothing compiles it in CI — so a
type error or an iOS-availability mistake only surfaces when someone next opens
Xcode on a Mac.

The obvious "run xcodebuild on PR" doesn't work off the shelf, because of how
the project is tracked:

- The entire `ios/` directory is **generated by `cap add ios` and gitignored**
  (see `docs/ios-native-app.md`). Nothing about the Xcode project is in git.
- The Share Extension **target** was historically added by hand in the Xcode IDE
  ("this can't be scripted from Linux"), living only in a developer's local,
  uncommitted `.xcodeproj`.

So a naive `xcodebuild` against a freshly generated project would compile the
empty Capacitor shell and **never touch `ShareViewController.swift`** — a green
check that tests none of the code that varies.

## Approach

Make the whole project — including the extension target — reproducible from
committed sources, then build it.

Considered two mechanisms:

1. **XcodeGen** (`project.yml` regenerates the whole `.xcodeproj`). Rejected: it
   *replaces* Capacitor's project generation, forcing us to reproduce the App
   target's settings, web-asset copy phase, and Pods wiring by hand and keep
   them in sync across Capacitor upgrades.
2. **`xcodeproj` gem patch script** (chosen). *Augments* Capacitor's output: let
   `cap add ios` generate the project as today, then run a committed script that
   adds the extension target. Capacitor upgrades flow through untouched, and the
   script doubles as the scripted form of the manual Xcode steps in the docs —
   fixing local setup reproducibility too.

## Components

### `scripts/add-share-extension.rb`

Idempotent Ruby (uses the `xcodeproj` gem, which CocoaPods already depends on).
Given a generated `ios/App/App.xcodeproj`, it:

- Tears down any `ShareExtension` target/group/embed-phase/dependency from a
  prior run (so re-running is safe), then recreates them.
- Creates a `com.apple.product-type.app-extension` target named `ShareExtension`.
- References `native/ShareExtension/ShareViewController.swift` **in place** (a
  compile input) and `Info.plist` (navigator only) — `native/` stays the source
  of truth; nothing is copied into `ios/`.
- Sets build settings: `PRODUCT_NAME = $(TARGET_NAME)` (without it the wrapper is
  named `.appex` and two build commands collide), `PRODUCT_BUNDLE_IDENTIFIER`
  derived as `<app id>.ShareExtension`, `INFOPLIST_FILE` pointing at the
  hand-authored plist, `GENERATE_INFOPLIST_FILE = NO`, `SWIFT_VERSION = 5.0`,
  and `IPHONEOS_DEPLOYMENT_TARGET = 26.0`.
- Wires `Secrets.xcconfig` as the target's base configuration (supplies
  `OTB_INGEST_API_KEY`); a missing file is a warning, not a failure.
- Adds the extension as a dependency of `App` plus an **Embed App Extensions**
  copy-files phase, so building the `App` scheme also builds the extension.
- Enables **Mac Catalyst** on both the App and extension targets, and raises the
  App target + project deployment target to match the extension.

**Deployment target = 26.0 across the whole app.** Capacitor generates the App
target at 13.0; the script raises the App target, project, and extension to 26.0
so the app has a single iOS floor rather than a 13/26 split. This is a
single-user app that runs the current iOS, so targeting the latest is fine — and
it clears the iOS 14 `UTType` API `ShareViewController.swift` relies on. (A
too-low floor was caught by the build during implementation: at 13.0 it fails
with "'UTType' is only available in iOS 14.0 or newer".)

### `.github/workflows/ios-build.yml`

- **Trigger:** `pull_request` **path-filtered** to `native/**`,
  `capacitor.config.ts`, the script, the workflow, `package.json`, `bun.lock`;
  plus `workflow_dispatch`. macOS runners bill at 10x Linux minutes, and the
  shell loads the live site, so web-only PRs don't need it.
- **Runner:** `macos-26`, Xcode `latest-stable`.
- **Steps:** checkout → select Xcode → setup Bun → **setup Ruby 3.3** →
  `gem install cocoapods xcodeproj` → `bun install` → placeholder
  `build/client/index.html` → dummy `Secrets.xcconfig` → `bun run cap:add` →
  `ruby scripts/add-share-extension.rb` → `xcodebuild … build` for **both** the
  iOS Simulator and **Mac Catalyst** destinations (`CODE_SIGNING_ALLOWED=NO`).
  The Catalyst build reuses the same generated project + Pods, so it's mostly
  incremental compile.

Two environment gotchas encoded in the workflow:

- **Ruby:** Capacitor's pod pre-flight (`checkBundler`) fails against the
  read-only system Ruby with `Gem::FilePermissionError`. A `ruby/setup-ruby`
  modern Ruby with a writable gem dir avoids it. (Reproduced locally; documented
  in the troubleshooting section too.)
- **No secrets:** a simulator build with `CODE_SIGNING_ALLOWED=NO` needs no
  certificates or provisioning profiles, and any `OTB_INGEST_API_KEY` value
  compiles — so the real ingest key never touches CI.

## Non-goals (YAGNI)

- **Pods/DerivedData caching.** Kept the first version simple; a caching pass is
  a straightforward later optimization if macOS minutes become a concern.
- **Code signing / device or TestFlight builds.** This is a compile check, not a
  distribution pipeline.

## Verification

Reproduced the full flow locally on macOS (Xcode 26.6): `cap add` → script →
`xcodebuild` with signing disabled, for **both** the iOS Simulator and Mac
Catalyst destinations. Confirmed the script is idempotent, that
`ShareViewController.swift` is genuinely compiled, and that `ShareExtension.appex`
is produced and embedded in `App.app` for both destinations.
